# ADR-003: Content Boundary Spotlighting for Untrusted Code

## Status
Accepted

## Context
codesight-mcp returns source code from indexed repositories to AI models via MCP tool responses. This source code is untrusted -- it may contain prompt injection payloads crafted to manipulate the AI into executing attacker instructions. Microsoft's spotlighting research demonstrated that boundary markers reduce prompt injection success rates from over 50% to under 2%.

## Decision
All untrusted content (source code, file paths, symbol names, signatures) returned by tool handlers is wrapped in boundary markers with cryptographically random tokens. The implementation lives in `src/codesight_mcp/core/boundaries.py`.

**Marker format** (not XML-style):
```
<<<UNTRUSTED_CODE_{token}>>>
{content}
<<<END_UNTRUSTED_CODE_{token}>>>
```

Where `token` is `secrets.token_hex(16)` (32 hex characters), generated fresh per `wrap_untrusted_content()` call. The random token makes it practically impossible for content to forge a matching end marker to "escape" the boundary.

**Application**: Tool handlers individually call `wrap_untrusted_content()` on each untrusted field. This is NOT applied centrally in `server.py`. The following tool modules apply wrapping (non-exhaustive):
- `search_symbols.py`, `search_text.py`, `search_references.py` -- search results
- `get_symbol.py`, `get_symbol_context.py` -- symbol source, signatures, docstrings
- `get_file_outline.py`, `get_file_tree.py`, `get_repo_outline.py` -- file paths and outlines
- `get_callers.py`, `get_callees.py`, `get_call_chain.py`, `get_impact.py` -- graph results
- `get_type_hierarchy.py`, `get_imports.py`, `get_dependencies.py` -- relationship data
- `get_changes.py`, `compare_symbols.py`, `get_dead_code.py` -- diff and analysis results
- `analyze_complexity.py`, `get_key_symbols.py`, `get_diagram.py` -- derived analysis
- `list_repos.py` -- repository names from the index

**Trusted metadata**: The `_meta` envelope (built by `make_meta()` in `boundaries.py`) carries `source`, `contentTrust`, and `warning` fields. These are NOT wrapped because they are generated by the server, not derived from indexed content.

## Consequences
**Positive**: AI models receive clear boundary signals distinguishing trusted server metadata from untrusted user content. Per-call random tokens prevent content from predicting and forging end markers.

**Negative**: Response payloads are slightly larger due to marker overhead. Each untrusted field gets its own token, which is deliberate -- a shared token per response would allow content in one field to close the boundary for another field.
